# Инструкции для Claude Code: разработка SEO-сайтов через вайбкодинг

> Плейбук для сборки SEO-ориентированных сайтов с нуля под управлением Claude Code.
> Формат рассчитан на использование как `CLAUDE.md` в корне проекта или как референс
> в отдельной папке. Все правила — обязательные, если не помечено как «опционально».

---

## 0. Назначение и зона применения

Плейбук покрывает разработку **контентных и услуговых SEO-сайтов** (лендинги услуг, образовательные порталы, корпоративные сайты, контент-проекты). Не покрывает: магазины с 10k+ SKU, B2B с ЛК и интеграцией с 1С — для них нужен отдельный шаблон.

**Философия**: строится не отдельный сайт под конкретную нишу, а **универсальный движок «сайт услуг»**, который адаптируется под любую нишу (автомойка, юристы, клининг, IT-агентство) сменой одного файла контента (`services.json`) и одного файла дизайн-токенов (`tailwind.config.ts`). Один шаблон → десятки сайтов. Claude Code в этой схеме — экзоскелет, ускоряющий работу в разы, а не «волшебная кнопка».

Главные требования к итоговому сайту:
- Контент должен быть в исходном HTML (SSG/SSR), а не подгружаться JS на клиенте
- PageSpeed ≥ 90 на десктопе, ≥ 80 на мобильных
- Уникальные `<title>`/`<meta description>` на каждой странице
- Sitemap, robots, 404, schema.org, hreflang (если многоязычно) — настроены автоматически
- Контент меняется через правку JSON, а не через правку компонентов

## 1. Definition of Done — критерии готовности сайта

Сайт не считается готовым, пока не пройден весь чек-лист. Применяется ко всем страницам перед боевым запуском. Без галочек по всем пунктам проект не релизится и не передаётся клиенту.

### Технические требования
- [ ] Консоль DevTools (F12) чистая — ноль ошибок и красных предупреждений
- [ ] Адаптив с 320px (Mobile First): нет горизонтального скролла на мобильных, корректное отображение на планшетах и десктопе
- [ ] Lighthouse / PageSpeed ≥ 90 (зелёная зона) на десктопе, ≥ 80 на мобильных
- [ ] Код отформатирован Prettier, без ошибок линтера, закоммичен в Git

### SEO-требования
- [ ] Уникальные `<title>` и `<meta description>` на каждой странице (через `generateMetadata`)
- [ ] Иерархия заголовков соблюдена: одна H1 → H2 → H3 (без пропусков уровней)
- [ ] У всех изображений заполнен `alt` (через поле `imageAlt` в JSON)
- [ ] ЧПУ-адреса: `/services/seo-audit`, не `/page?id=15`
- [ ] Автоматически сгенерированный `sitemap.xml`, доступный по `/sitemap.xml`
- [ ] `robots.txt` со ссылкой на sitemap, открытый для всех юзер-агентов
- [ ] Open Graph теги (`og:title`, `og:description`, `og:image`) + Favicon
- [ ] Schema.org JSON-LD на ключевых страницах (Organization, Service, FAQPage)

### Функциональные требования
- [ ] Формы отправляют данные и доходят до получателя
- [ ] Телефоны кликабельны (`tel:`), мессенджеры открываются (`https://wa.me/...`, `https://t.me/...`)
- [ ] Кастомная страница 404 со ссылкой «На главную» — не дефолтная страница сервера
- [ ] HTTPS работает, есть редирект http → https

Этот чек-лист — обязательная часть передачи проекта. Контролируй Claude Code по нему: после реализации каждой страницы прогоняй пункты явно.

## 2. Пайплайн разработки: 10 этапов

Проекты «умирают» из-за нарушения порядка действий — например, вёрстки без готовой структуры или дизайна без реального контента. Соблюдай последовательность:

| № | Этап | Суть | Что делает Claude Code |
|---|---|---|---|
| 1 | **Архитектура и структура** | Дерево страниц до кода (Главная, Каталог, Услуга, Спасибо, Контакты) | Помогает составить карту сайта в Markdown / Mermaid |
| 2 | **Дизайн и UI-кит** | Палитра, шрифты, отступы, виды кнопок — правила игры до вёрстки | Извлекает токены из Figma MCP или референса в `tailwind.config.ts` |
| 3 | **Каркас проекта (Setup)** | Инициализация Next.js, Tailwind, TypeScript strict, линтеры, `CLAUDE.md` | Скаффолдинг по правилам из § 4 |
| 4 | **Роутинг и контент (Content First)** | Сначала JSON с реальным контентом, потом URL и вёрстка | Заполняет `services.json` и поднимает `/services/[slug]` |
| 5 | **Блоки и шаблоны** | Конструктор Lego: один FAQ-блок работает на главной и на услугах | Создаёт `BlockRenderer` и блоки в `/components/blocks` |
| 6 | **Техническое SEO** | Метатеги и семантика **во время разработки**, а не перед релизом | `generateMetadata`, schema.org, семантический HTML5 |
| 7 | **Скорость и Web Vitals** | WebP, lazy-loading, оптимизация шрифтов и JS | `next/image priority`, `next/font`, `next/dynamic` |
| 8 | **Аналитика** | Метрика / GA + цели на конверсии | `next/script afterInteractive` + `reachGoal` на успешный submit |
| 9 | **Деплой (Релиз)** | DNS, SSL, HTTPS, прогон чек-листа на проде | systemd + Nginx + certbot или Vercel |
| 10 | **Поддержка** | Правка цен, телефонов, текстов через JSON за 5 минут, без HTML | Архитектура двигателя блоков (§ 6) |

**Принципы пайплайна**:
- **Content First**: верстать вокруг реального контента, не вокруг lorem ipsum — при наполнении вёрстка разваливается на «слишком длинных» заголовках или кейсах с 8 пунктами вместо 3
- **SEO во время разработки**: метаданные и семантика встраиваются на этапе 6, а не «когда сайт готов» — иначе придётся переписывать компоненты
- **Поддержка через конфиг**: если для смены цены нужно править HTML — архитектура сломана, возвращайся к этапу 5

## 3. Технический стек (обязательный)

```
Framework:   Next.js 15 (App Router)
Language:    TypeScript (strict)
Styling:     Tailwind CSS
Routing:     встроенный App Router
SEO:         встроенный generateMetadata + generateStaticParams
Контент:     JSON-файлы в /src/data/
Аналитика:   Yandex.Metrika через next/script strategy="afterInteractive"
```

Запрещено без явного запроса: Chakra UI, Material UI, Redux, styled-components, любые CSS-in-JS-рантаймы, классический WordPress (не headless), Webpack-сборка.

## 4. Файловая структура

```
/src
  /app                     # ТОЛЬКО роутинг и серверные страницы
    layout.tsx
    page.tsx
    not-found.tsx
    robots.ts
    sitemap.ts
    /services
      page.tsx             # каталог услуг (hub)
      /[slug]
```

**ЗАПРЕЩЕНО** создавать компоненты в корневой папке `/src` или прямо в `/app`. Серверные страницы импортируют блоки из `/components/blocks`, а не определяют их локально.

## 6. Архитектура: «Двигатель блоков»

### Принцип

Контент **не хардкодится** в JSX. Каждая страница описывается массивом блоков в JSON, рендерер обходит массив и вызывает соответствующий компонент.

### Модель данных

`src/data/services.json` — каждый объект описывает одну услугу:

```json
{
  "id": 1,
  "slug": "seo-audit",
  "title": "SEO-аудит сайта",
  "shortDescription": "Карточка для каталога",
  "metaTitle": "SEO-аудит сайта — 2GIGA Agency",
  "metaDescription": "Полный технический и контентный аудит за 5 дней",
  "image": "/images/seo-audit.jpg",
  "imageAlt": "Скриншот отчёта SEO-аудита",
  "blocks": [
    {
      "type": "HeroBlock",
      "data": {
        "title": "Заголовок",
        "description": "Подзаголовок",
        "image": "/images/seo-main.png",
        "imageAlt": "Описание для alt",
        "ctaText": "Подробнее",
        "ctaHref": "/#start"
      }
    },
    { "type": "FAQBlock", "data": { "title": "Вопросы", "items": [] } },
    { "type": "ReviewsBlock", "data": { "title": "Отзывы" } }
  ]
}
```

Связи «многие-ко-многим» (например, отзывы ↔ услуги) — через массив id:
```json
// reviews.json
{ "id": 1, "text": "...", "serviceIds": [1, 3] }
```

### BlockRegistry и BlockRenderer

`src/components/blocks/BlockRenderer.tsx` — словарь `type` → компонент:

```ts
const BlockRegistry = {
  HeroBlock,
  FAQBlock,
  ReviewsBlock,
  // добавление блока = одна строка
} as const;

type BlockProps = { blocks: Block[]; context?: { serviceId?: number } };
```

Рендерер обходит массив `blocks`, ищет компонент в реестре, передаёт `data` и `context` как props.

### Правило

Чтобы добавить новый тип блока:
1. Создать компонент в `/components/blocks/NewBlock.tsx`
2. Импортировать в `BlockRenderer.tsx` и добавить в `BlockRegistry`
3. Использовать `"type": "NewBlock"` в JSON

Никогда не вшивать тексты прямо в компонент — только через `props.data`.

## 7. Роутинг и динамическое SEO

### Структура

- `app/services/page.tsx` — каталог-хаб со списком услуг
- `app/services/[slug]/page.tsx` — динамическая страница услуги
- `app/not-found.tsx` — кастомная 404 (отдаёт корректный HTTP 404, не soft-200)

### Обязательные функции Next.js

**generateStaticParams** — пререндер всех страниц при сборке:
```ts
export async function generateStaticParams() {
  return services.map(s => ({ slug: s.slug }));
}
```

**generateMetadata** — уникальные title/description из данных:
```ts
export async function generateMetadata({ params }) {
  const service = services.find(s => s.slug === params.slug);
  if (!service) return {};
  return {
    title: service.metaTitle,
    description: service.metaDescription,
    alternates: { canonical: `/services/${service.slug}` },
    openGraph: { title: service.metaTitle, description: service.metaDescription, images: [service.image] }
  };
}
```

**notFound()** — защита от soft-404:
```ts
const service = services.find(s => s.slug === params.slug);
if (!service) notFound();
```

### Запрет

Не использовать `useEffect` для подгрузки контента, который должен индексироваться. Всё, что нужно боту, рендерится на сервере (Server Components по умолчанию в App Router).

## 8. Каркас: Layout, Header, Footer

Единый `components/layout/Layout.tsx` подключается из `app/layout.tsx`:

```tsx
<div className="flex min-h-screen flex-col">
  <Header />
  <main className="flex-1">{children}</main>
  <Footer />
</div>
```

**Header**: семантический `<nav>`, sticky-поведение (`sticky top-0 z-50`), активное состояние текущей страницы, мобильное меню с областью нажатия ≥ 44×44 px.

**Footer**: семантический `<footer>`, три колонки минимум (О компании / Быстрые ссылки / Контакты), копирайт. Это сигнал доверия (E-E-A-T) и точка распределения ссылочного веса.

**Контрастность**: проверять Lighthouse-ом, текст на тёмном фоне — светлый, минимум WCAG AA.

## 9. Дизайн-пайплайн

1. **Получение визуала**: Stitch AI / v0 / Figma. Если макет в Figma — использовать MCP-инструмент `get_design_context(fileKey, nodeId)`, получить React+Tailwind-черновик и скриншот для сверки.

## 10. Индексируемость и проверки рендеринга

### Два обязательных теста перед каждым запуском

**Тест 1. View Source (Ctrl+U)**
1. Открыть страницу
2. Ctrl+U → Ctrl+F → найти текст H1
3. **Успех**: текст виден в сыром HTML
4. **Провал**: пустой `<div id="root">` — переписать компонент на серверный рендеринг

**Тест 2. Disable JavaScript**
1. F12 → Ctrl+Shift+P → Disable JavaScript
2. Обновить страницу
3. **Успех**: контент виден, ссылки работают, картинки на месте
4. **Провал**: белый лист или пропавший контент — убрать `useEffect`-загрузку, перевести на серверный компонент

### Запрет на CSR для контента

Любой текст, заголовок, картинка или ссылка, которые нужны SEO, рендерится на сервере. Клиентские компоненты (`'use client'`) допустимы только для интерактива (формы, аккордеоны, модалки), не для контентных блоков.

## 11. Sitemap, robots, technical SEO

### sitemap.ts

`app/sitemap.ts` — автогенерация из JSON. Прохождение по `services.json` через `.map()`, фильтр черновиков:

```ts
const servicesUrls = services
  .filter(s => !s.draft)
  .map(s => ({
    url: `${BASE_URL}/services/${s.slug}`,
    lastModified: s.updatedAt,
    changeFrequency: 'weekly',
    priority: 0.8
  }));
```

Для черновиков добавлять в JSON `"draft": true`.

### robots.ts

`app/robots.ts`:
- `Allow: /` для всех юзер-агентов
- `Disallow: /api/`, `/admin/` (если есть)
- ссылка на `sitemap.xml`

### Disallow vs Noindex (важная разница)

- **Disallow в robots.txt** — запрет на сканирование. Если на страницу ведут внешние ссылки, Google всё равно может показать её в выдаче с пустым сниппетом.
- **Noindex в `<meta>`** — запрет на индексацию. Чтобы бот увидел этот тег, путь к странице **должен быть открыт в robots.txt**, иначе он туда не дойдёт.

Скрытие страницы из выдачи = `Allow` в robots + `<meta name="robots" content="noindex">` в `<head>`.

### Schema.org

На каждой странице услуги — JSON-LD `Service` или `Product`. На FAQ-блоке — `FAQPage`. На главной — `Organization`. Хлебные крошки — `BreadcrumbList`.

### Канонические URL

`alternates.canonical` в `generateMetadata` на каждой странице. Self-referencing для обычных, страница без параметров для пагинированных.

### Анти-дубли

Middleware (`middleware.ts`) приводит URL к канонической форме:
- `/PAGE/` → `/page/` (lowercase)
- `www.example.ru` → `example.ru`
- `/index.html` → `/`
- `?utm_*` не создаёт новый canonical

## 12. Производительность (Core Web Vitals)

### Целевые метрики

| Метрика | Зелёная зона | Решение |
|---|---|---|
| **LCP** (Largest Contentful Paint) | < 2.5 с | `<Image priority>` для главного, WebP, фиксированные размеры |
| **CLS** (Cumulative Layout Shift) | < 0.1 | жёсткие width/height, `next/font` с `display: 'swap'` |
| **INP** (Interaction to Next Paint) | < 200 мс | `next/dynamic` для тяжёлого JS, удаление синхронных скриптов |

### Обязательные правила

**Изображения**: только `next/image`, не голый `<img>`.
- Для первого экрана: `priority={true}`
- Для остальных: ленивая загрузка по умолчанию
- Всегда указывать `width` и `height`
- `alt` обязателен (брать из `imageAlt` в JSON)

**Шрифты**: только `next/font/google`, никаких `<link href="fonts.googleapis.com">`.
```ts
import { Inter } from 'next/font/google';
const inter = Inter({ subsets: ['latin', 'cyrillic'], display: 'swap' });
```

**Тяжёлый JS**: вынос в асинхронную загрузку.
```ts
const Calculator = dynamic(() => import('./Calculator'), { ssr: false });
```

**Сторонние скрипты** (Метрика, GA): через `next/script` с `strategy="afterInteractive"`, не в `<head>`.

## 13. Аналитика

### Подключение счётчика

В `app/layout.tsx`:
```tsx
<Script
  id="ym"
  strategy="afterInteractive"
  dangerouslySetInnerHTML={{ __html: `/* код Метрики */` }}
/>
```

Никогда не вставлять скрипт в `<head>` напрямую — это блокирует рендеринг и убивает LCP.

### Цели на конверсии

Все вызовы `ym(...)` оборачивать в проверку — у клиента может быть AdBlock:
```ts
if (typeof window !== 'undefined' && typeof window.ym !== 'undefined') {
  window.ym(COUNTER_ID, 'reachGoal', 'click_phone');
}
```

**Где вешать**:
- Клик по телефону / Telegram / WhatsApp — на `onClick`
- Отправка формы — **только на успешный ответ** (после успешного `fetch`), не на клик кнопки. Иначе попадут заявки с ошибкой валидации.

### Отладка

Дописать в URL `?_ym_debug=1`, открыть консоль (F12) — увидеть отправку целей.

### Какие отчёты смотреть SEO-специалисту

- **Источники → Поисковые системы**: динамика Яндекс vs Google, резкое падение = фильтр
- **Содержание → Страницы входа**: высокий «Отказ» (>30%) = страница нерелевантна запросу
- **Технологии → Устройства**: разница конверсии ПК / Смартфон = сломанная мобильная вёрстка
- **Вебвизор**: запись экрана с фильтром «Отказники» (< 15 с) — UX-проблемы

## 14. Деплой на VPS (Ubuntu)

> Альтернатива: Vercel (бесплатно до 100k визитов, push в GitHub → автодеплой). Используется, если клиент не требует свой VPS.

### Шаги

1. **SSH**: `ssh root@IP_сервера`. Базовые команды: `pwd`, `ls`, `mkdir`, `cd`, `vim`.
2. **Копирование**: `scp -r ./* root@IP:/root/имя-проекта`.
3. **Сборка**: на сервере — `npm install`, затем `npm run build`.
4. **Systemd-демон**: создать `/etc/systemd/system/имя-сайта.service` для автозапуска, выполнить `systemctl enable` + `systemctl start`. Конфиг просить у Claude Code: «напиши systemd-юнит для Next.js на порту 3000».
5. **Nginx**: установить, написать конфиг (проксирование `:80` → `:3000`). Промпт: «напиши конфиг Nginx для проксирования с домена example.ru на localhost:3000».
6. **SSL**: `certbot --nginx -d example.ru` — Let's Encrypt сертификат бесплатно, настройки HTTPS прописываются автоматически.

### Чек перед боевым релизом

## 17. Работа с AI-правками: правило diff

ИИ способен «галлюцинировать» — удалить нужную логику, переписать рабочий компонент, потерять обработчик ошибок. Поэтому **никогда не принимать правки не глядя**.

Перед тем как принять правки от Claude Code:
- **Красные строки** — то, что **удаляется**. Прочитать каждую: точно ли это лишнее? Не удаляется ли вместе с дублем нужный обработчик?
- **Зелёные строки** — то, что **добавляется**. Проверить логику, типы, импорты — соответствует ли архитектуре проекта (§ 4) и конвенциям (§ 5)?
- **При сомнении — отклонить и попросить переделать** с уточнением задачи. Принимать правки только когда понятно, что именно меняется и почему.

После каждой партии изменений:
1. Прогнать линтер и type-check (`npm run lint`, `tsc --noEmit`)
2. Проверить, что страница открывается без ошибок в консоли
3. Закоммитить мелкими атомарными коммитами — чтобы откат стоил минуту, а не день

## 18. Что НЕ делать

- ❌ Хардкодить тексты в JSX. Только через `props.data` из JSON.
- ❌ Подгружать контент через `useEffect` или клиентский `fetch` для SEO-страниц.
- ❌ Создавать компоненты в корне `/src` или `/app`.
- ❌ Использовать классические `<img>`, `<link href="fonts.">`, синхронные сторонние скрипты.
- ❌ Вставлять скрипты аналитики в `<head>` — только через `next/script afterInteractive`.
- ❌ Менять slug у проиндексированных страниц без 301-редиректа.
- ❌ Полагаться на «пустую» 404 (soft 404) — всегда `notFound()` + кастомный `not-found.tsx`.
- ❌ Игнорировать тест Disable JavaScript перед релизом.
- ❌ Принимать AI-правки не читая diff — ИИ ошибается и удаляет рабочий код.
- ❌ Релизить сайт без полного прохождения чек-листа Definition of Done (§ 1).
